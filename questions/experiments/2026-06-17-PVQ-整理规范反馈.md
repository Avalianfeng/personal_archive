## 2. 整理规范-v1.0 好用之处

### 2.1 流程极简、可执行

§ Agent 整理产出 + § 批次验收四步命令，比旧版 raw→categories→build_questions 链短很多。Agent 只需写 jsonl、跑四条命令，职责边界（不负责 uid/id）清晰。

### 2.2 DB 时代禁止项明确

「禁止手改 categories」「禁止 ingest 扫描 source_library」两条在实验中直接避免了常见误操作；子分类通过 jsonl `subcategory` 字段自动 sync，本批新增「个人价值观」无需碰 md。

### 2.3 与卡片协作良好

`[待整理][P2] PVQ.md` 的整理注意（第三人称改写、agreement/rating、6 级锚点）与规范不冲突，足够完成本批。

### 2.4 批次验收清单可核对

§21 的「pending 为空」是硬指标，实验结束后可客观确认归档成功。

## 3. 缺口与矛盾

### 3.1 options 格式：三处不一致（**阻塞级**）

| 文档 | 写法 |
| --- | --- |
| **整理规范-v1.0 示例** | 无 `options` 字段 |
| **本实验任务说明示例** | `"options":["完全不像我",…]` 字符串数组 |
| **`schema/question.schema.json`** | `[{"key":"…","text":"…"}]` 对象数组 |
| **`sync_categories.py`** | 假定 `opt['key']` / `opt['text']` |

`ingest.py` **接受**字符串数组并写入 DB，但 **sync 必崩**。首跑 ingest 已 commit 40 题，sync 失败，doctor 报 categories hash drift——处于「半成功」状态，Agent 需额外排查。

### 3.2 问题整理提示词仍为 v0.4 / categories 时代

提示词正文仍描述：

- 输出「按分类分块的 Markdown frontmatter」
- `build_questions.py` · `--strict-registry` · `--audit-dyadic`
- 流程图 `raw/pending/ → categories/*.md`
- 末尾链接 **`整理规范-v0.1.md §二十一`**（v1.0 仅 §21，无「二十一」中文节号）

与 v1.0「jsonl → ingest」直接冲突；Agent 若只读提示词会走错路径。

### 3.3 格式规范.md 未同步到 DB 时代

仍含 `raw/pending`、`question_sources/`、`build_questions.py` 编译链；Agent 导入契约一行带过 `options` 语义。读此文件无法得知 agreement 量表 options 写法。

### 3.4 分类原则.md 链接旧版

引用 `整理规范-v0.1.md §十三`，v1.0 已无对应章节。

### 3.5 type 命名：提示词 vs 卡片 vs schema

提示词表格写「量表端点完整 → `scale` 或 `agreement`」；PVQ 卡片写「认同度或量表(6)」。无决策树说明何时用 `agreement` vs `scale`、是否必填 `options` vs `scale_note`。

### 3.6 ingest 失败时的恢复路径未文档化

规范只说「若 ingest 失败，读 error log 修复 jsonl 后重试」，未说明：

- sync 在 ingest 末尾自动触发，**DB 已写入但 sync 失败**时如何处理
- 是否应 `manage sync` 单独重跑，还是删库重 ingest

| 步骤 | 摩擦 |

## 5. 具体改进建议

### P0（必须立即修）

1. **`整理规范-v1.0.md` 增补 jsonl 字段表**  
   必填：`question, category, subcategory, type, source`；建议：`interaction, status, options`。  
   **明确**：`options` 必须为 `[{"key":"1","text":"…"},…]`，并给出 agreement 6 级完整示例（可引用 PVQ）。

2. **统一实验任务说明 / 提示词中的 options 示例**  
   凡出现 `"options":["…"]` 字符串数组处改为对象数组，或注明「ingest 暂容字符串但 sync 会失败」。

3. **`问题整理提示词.md` 升级到 DB/jsonl 时代（至少正文输出格式节）**  
   - 删除/标注废弃：Markdown 分块输出、build_questions、raw 流程图  
   - 输出格式改为 jsonl 每行示例（含 options 对象）  
   - 批次验收链接改为 `整理规范-v1.0.md §批次验收`

4. **`ingest.py` 入库前校验 options 形状**（或在 `insert_question_record`）  
   与 schema 一致则拒绝字符串数组，错误信息指向规范 §options，避免「入库成功、sync 崩溃」。

### P1（下一迭代）

5. **`schema/格式规范.md` 瘦身并重写 Agent 契约节**  
   删除 raw/build 段落；新增 agreement/scale/single 的 options vs scale_note 对照表。

6. **整理规范增补「量表类来源整理 checklist」**  
   - 第三人称→第二人称  
   - type=agreement + interaction=rating  
   - options 6 级 key 用 `"1"`…`"6"` 还是 `"A"`…`"F"`  
   - 整批同一 subcategory 还是按 Schwartz 维度拆分（本实验选前者）

7. **分类原则「价值问题」例题增加量表条目**  
   如「个人价值观 · PVQ 式 portrait 陈述 + 像我量表」。

8. **ingest 事务边界**  
   sync/export 失败时回滚或明确文档化「仅重跑 manage sync」路径。

9. **修正交叉引用**  
   `分类原则.md`、`问题整理提示词.md` 中 v0.1 / §二十一 / §十三 链接改为 v1.0 对应节。

### P2（体验优化）

10. **子分类命名指南**  
    量表整批：「个人价值观」vs 拆「优先级/底线」；新增子分类是否与分类原则例题对齐。

11. **source_library 卡片模板**  
    整理记录表（题数、id 范围、processed 文件名）可列为卡片标准字段，本实验已手写。

12. **`qcli doctor` 区分「sync 未跑」与「数据错误」**  
    首跑后 hash drift 警告可提示「运行 manage sync」。

## 6. 对 `问题整理提示词.md` 是否应同步升级

**应，且优先级 P0。**

当前提示词是 v0.4 categories 时代的「主操作手册」，而 v1.0 规范已是「主规范」。实验中的实际路径 100% 来自 v1.0 + 任务说明；提示词仅在 system_unaskable、type 表、slug 对照上有用，**输出格式与验收链已误导**。

建议：

- 保留：去格式原则、system_unaskable 表、category slug 表、type/interaction 枚举、翻译与残缺表  
- 重写：§输出格式 → jsonl；§仓库内流程 → imports/pending → ingest → sync → duplicate_scan → doctor  
- 删除或移入「历史附录」：frontmatter 示例、ID 规则（ingest 自动生成）、build_questions 引用  
- 整理员阅读笔记可保留，并追加本实验条目（options 对象格式、PVQ 子分类）

## 9. 结论

整理规范-v1.0 的 **DB 流程骨架正确且比旧流程轻量**，本批 PVQ 可在遵循 jsonl 契约的前提下完成入库与验收。  
**最大缺口是 options 格式与配套文档未对齐**，导致首跑「ingest 看似成功、sync 必挂」——这是规范、提示词、schema、任务示例四者之间的 **P0 级矛盾**，应在下一版规范与提示词升级中优先消除。
